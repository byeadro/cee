# 15 — SKILL FILE STRUCTURE
> **Status:** Authoritative · **Owner:** AB · **Reads on boot:** Yes
> **Purpose of this page:** the byte-level specification for [SKILL.md](http://SKILL.md) files. Section 07 defined the Skill *system* — matching, generation, promotion. This page defines the *file* — exact frontmatter schema with allowed value types, body section conventions, supporting file rules, naming rules, and the worked examples that anchor every other reference. If section 07 is the engine, this page is the data sheet.
---
## 1. What This Is
A Skill in CEE is a directory: `~/cee/skills/<slug>/` containing exactly one required file (`SKILL.md`) and zero or more optional supporting files. The required file has YAML frontmatter followed by a markdown body. This page nails down:
- The exact frontmatter schema (every field, every type, every constraint)
- The naming convention for slugs and supporting files
- The body structure (which sections are required, recommended, or optional)
- The supporting file types (`examples.md`, `references.md`, `_meta.json`)
- The size and length budgets
- Worked examples for each task_type
- The validators that enforce all of the above
This page is the source the Skill schema (`~/cee/schemas/skill_frontmatter.json`) is derived from. If the schema and this page disagree, this page wins and the schema gets updated.
---
## 2. Why This Matters
Without an exact spec:
- Skills generated by CEE drift over time as the generator's prompt evolves.
- Hand-authored Skills (the catalog seeds, OPERATOR-written ones) deviate in ways that break the matcher.
- Schema validation passes but the matcher fails because `description` quality varies.
- New OPERATORs (or future-AB returning to the project) can't tell what good looks like.
The exact spec makes "valid Skill" a property the validator can check programmatically. Generation drift is caught at write time, not at runtime when matching fails. Worked examples make "good Skill" a thing OPERATORs can imitate.
---
## 3. Core Requirements
A Skill file MUST:
1. Live at `~/cee/skills/<slug>/SKILL.md`.
2. Open with a YAML frontmatter block (`---` delimited) containing every required field per §5.2.
3. Have a body conforming to the section structure in §5.3.
4. Be valid markdown (no broken syntax).
5. Validate against `~/cee/schemas/skill_frontmatter.json`.
6. Have a body whose total length is within the budgets in §5.5.
7. Be self-contained — references to other Skills, agents, or external resources are explicit and non-required.
A Skill file MUST NOT:
- Have a slug that violates the naming rules in §5.1.
- Have frontmatter with extra fields not in the schema (Pydantic strict mode rejects).
- Use Claude Code-internal tool names that don't exist (e.g., referencing a deprecated tool).
- Embed CEE-internal jargon (the Skill is loaded by the executor at execution time; the executor doesn't know CEE exists).
- Have the same slug as an agent (cross-namespace collision is forbidden).
---
## 4. System Rules
**Rule 1 — Slugs are kebab-case ASCII.**
Allowed characters: lowercase a-z, digits 0-9, hyphens. No underscores, no uppercase, no Unicode. Length 3–60 characters. Must start with a letter, end with a letter or digit.
**Rule 2 — One slug, one directory, one **[**SKILL.md**](http://SKILL.md)**.**
The directory is the slug. The file is exactly `SKILL.md` (case-sensitive). One Skill per directory.
**Rule 3 — Versioning is semver, in frontmatter and in directory name.**
Major version bumps create a new directory (`<slug>-v2/`). Minor and patch versions stay in the same directory; the frontmatter `version` field is the canonical version.
**Rule 4 — Frontmatter is the matcher's fuel.**
The matcher (per section 07 §5.2) reads `description`, `triggers`, `task_types_supported`, and `domain` from frontmatter. Body content is only loaded by the executor at execution time, not by the matcher. So frontmatter quality determines reuse rate.
**Rule 5 — The body is for the executor, not the human.**
The body is what Claude Code sees when it loads the Skill at execution time. Write for that audience. Avoid CEE-system commentary; focus on how to do the task.
**Rule 6 — Required sections in body.**
Every Skill body has: a one-paragraph "When to use" section, an "Instructions" section, and at least one "Example" section. Other sections are optional.
**Rule 7 — Length budgets.**
Frontmatter ≤ 50 lines. Body ≤ 800 words for typical Skills, ≤ 1500 for complex ones. Total file ≤ 2000 lines as a hard ceiling. Enforced by validator.
**Rule 8 — Supporting files are optional and limited.**
Allowed siblings: `examples.md`, `references.md`, `_meta.json`. Other filenames are reserved for future use; validator warns. Size of all supporting files combined ≤ 50KB (large reference material should be linked, not embedded).
**Rule 9 — Provenance is required for generated Skills.**
A generated Skill (`created_by_run` is a run_id) must include `created_at` and `created_from_input` (the verbatim user input that triggered generation, redacted per section 12). Hand-authored Skills (`created_by_run: manual`) have no required provenance fields beyond `created_at`.
**Rule 10 — Updates require a body changelog.**
A Skill's body has a final `## Changelog` section. Each version bump adds an entry: `### v1.1.0 — 2026-04-30: <one-line description of change>`. Validator warns if version bumped but changelog not updated.

## 5. Detailed Workflow — The File Spec
### 5.1 Slug naming rules
```javascript
slug ::= /^[a-z][a-z0-9-]{1,58}[a-z0-9]$/
```
Allowed:
- `read-codebase`
- `write-rls-policies`
- `next-app-router-page`
- `summarize-legal-doc`
Disallowed:
- `Read-Codebase` (uppercase)
- `read_codebase` (underscore)
- `read codebase` (space)
- `r` (too short)
- `1-fix` (starts with digit)
- `read-` (ends with hyphen)
Major version migration: `<slug>-v2`, `<slug>-v3`, etc. Original retains its slug; v2 is a new Skill that can be matched independently.
### 5.2 Frontmatter schema (full)
```yaml
---
# REQUIRED FIELDS

name: <slug>                          # must match parent directory name
description: |                        # YAML block scalar; one paragraph
  <natural-language description, 50-300 words; tells the matcher when to use this Skill>
version: <semver>                     # e.g., "1.0.0", "2.3.1"
triggers:                             # 1-10 phrases; matcher uses these
  - <natural-language trigger phrase>
  - <another trigger phrase>
inputs:                               # 1-10 short input descriptors
  - <input type or shape>
outputs:                              # 1-10 short output descriptors
  - <output type or shape>
task_types_supported:                 # 1-8 task_types from the closed enum
  - BUILD                             # any subset of: BUILD, ANALYZE, DEBUG, WRITE, RESEARCH, TRANSFORM, DECIDE, ORCHESTRATE
created_at: <ISO 8601 timestamp>      # creation time
created_by_run: <run_id> | manual | seed   # provenance

# OPTIONAL FIELDS

posture_hints:                        # 0-5 postures the Skill is well-suited for
  - primary                           # any subset of: primary, critic, optimizer, orchestrator, specialist
domain: <domain>                      # one of: code, writing, analysis, research, ops, personal, other
created_from_input: <verbatim text>   # required if created_by_run is a run_id
sensitivity: low | medium | high      # if high, forces flags.sensitive_data on Runs using this Skill
grounding_required: <bool>            # if true, forces flags.needs_grounding
disabled: <bool>                      # if true, registry skips this Skill
needs_review: <bool>                  # if true, Skill is generated and unreviewed; promotion pending
deprecated_at: <ISO timestamp>        # if set, Skill is deprecated; replacement_slug recommended
replacement_slug: <slug>              # the new Skill that supersedes this one (if deprecated)
notes: |                              # OPERATOR notes; not used by matcher
  Free-form notes about why this Skill exists, gotchas, history.
---
```

### 5.3 Body structure
```markdown
# <Skill Name>

<One-paragraph statement: what this Skill does, in one paragraph. The matcher doesn't read this, but humans skimming the file want it.>

## When to Use

<Two to four sentences describing the situations where this Skill applies. This expands on the description and the triggers list.>

## Inputs

<Detail what the executor should expect to receive. Reference the `inputs` field but expand: what file shapes, what sizes, what assumptions.>

## Outputs

<Detail what the executor produces. Reference the `outputs` field; expand on the structure.>

## Instructions

<The actual how-to. Step-by-step where the work has structure, prose where it doesn't. This is what Claude Code reads at execution time.>

### Step 1 — <action>
<details>

### Step 2 — <action>
<details>

(or, for unstructured Skills, plain prose)

## Example

<At least one worked example. Format:>

**Input:** <example input>

**Output:** <example output>

**Notes:** <any commentary about the example>

## Example 2 (optional)

<another worked example>

## Tools (optional)

<If the Skill assumes specific Claude Code tools, list them: Read, Edit, Bash, Write, Glob, Grep. State which tools are essential and why.>

## Failure modes (optional)

<If the Skill has known failure modes, document them. E.g., "this Skill assumes UTF-8 input; binary files will fail.">

## Related Skills (optional)

<Wiki-style links to other Skills that compose well: `[[other-skill-slug]]`. The executor doesn't auto-load these; the link is informational.>

## Changelog

### v1.0.0 — <date>: Initial creation.
### v1.1.0 — <date>: Added <feature>.
```
Required sections: top-level paragraph, `When to Use`, `Inputs`, `Outputs`, `Instructions`, at least one `Example`, `Changelog`.
Optional sections: additional `Example`s, `Tools`, `Failure modes`, `Related Skills`, plus any other H2 the OPERATOR adds.
### 5.4 Supporting files
#### `examples.md` (optional)
Additional worked examples that don't fit in `SKILL.md`. Same example format as the body's `Example` section. Used when:
- The Skill has many edge cases worth documenting.
- Examples are large (full file inputs/outputs).
The matcher does not read `examples.md`. Only the executor loads it if Claude Code's Skill mechanism does (as of current docs, supporting files are accessible via Read).
#### `references.md` (optional)
Links to external sources, papers, docs that informed the Skill. Format:
```markdown
# References for <skill name>

## Documentation
- [Anthropic Docs on Tool Use](https://docs.anthropic.com/...)

## Related work
- "Paper Title" — Author Year — URL

## Internal references
- Bible section: [[14 — CLAUDE CODE INTEGRATION]]
- Related Skill: `~/cee/skills/<other-slug>/`

## Notes
<free-form research notes>
```

### 5.6 Worked examples
Three Skills shown in full to anchor the spec.
#### Example 1 — `read-codebase` (BUILD/ANALYZE/DEBUG, code domain)
```markdown
---
name: read-codebase
description: |
  Walk a project directory, identify the entry points, key modules, and architecture. Used as the first step in BUILD, ANALYZE, or DEBUG tasks against an existing codebase. Produces a mental map: what the project does, how it's organized, what conventions it follows.
version: 1.0.0
triggers:
  - read this codebase
  - understand the project structure
  - map the architecture
  - find the entry points
  - what does this project do
inputs:
  - filesystem path to a project root
outputs:
  - structured summary of architecture, entry points, conventions
task_types_supported:
  - BUILD
  - ANALYZE
  - DEBUG
posture_hints:
  - primary
domain: code
created_at: 2026-04-30T00:00:00Z
created_by_run: seed
---

# Read Codebase

Walks a project directory to build a working understanding before any other work happens. Almost every meaningful BUILD, ANALYZE, or DEBUG task on an existing codebase starts with this Skill.

## When to Use

Use whenever you've been pointed at a project root and need to do anything beyond a one-line change. Skip for trivial single-file edits where context is unnecessary.

## Inputs

- A filesystem path to the project root.
- Optional: a focus area (e.g., "auth flow", "the API layer") to narrow attention.

## Outputs

A structured summary covering:

- **What it is:** one-paragraph project description, inferred from README / package.json / pyproject.toml.
- **Entry points:** how the project starts (main, CLI command, server entry).
- **Key modules:** the 5–10 most important files or directories, with one-line descriptions.
- **Conventions:** naming, file organization, test patterns, framework idioms.
- **Anything notable:** unusual choices, big abstractions, custom DSLs.

## Instructions

### Step 1 — Skim the root
Read README.md, package.json or pyproject.toml or Cargo.toml or equivalent, .gitignore, the top-level directory listing.

### Step 2 — Identify language and framework
From the manifest files. Note version constraints.

### Step 3 — Find entry points
Look for: `main.py`, `index.ts`, `cli.py`, `server.py`, scripts in package.json, `bin/`.

### Step 4 — Walk key directories
For each top-level source directory, list its contents at one level deep. Read the most-likely-important files (config files, app initializers, route definitions).

### Step 5 — Identify conventions
Note naming patterns (snake_case vs camelCase), file organization (feature folders vs type folders), test colocation.

### Step 6 — Produce the summary
Write the structured summary in the format above. Keep it under 500 words.

## Example

**Input:** `~/projects/embra-mentors/`

**Output:**

> **What it is:** Embra Mentors is a Next.js application connecting law students with attorney mentors. Production deploy at embra-mentors.vercel.app.
>
> **Entry points:** `app/page.tsx` (App Router root); `app/api/*` for serverless API routes; `scripts/seed.ts` for dev data.
>
> **Key modules:**
> - `app/(auth)/` — authentication routes (Supabase Auth)
> - `app/(marketing)/` — public pages
> - `app/(app)/` — authenticated app surface
> - `lib/supabase/` — Supabase client wrappers
> - `lib/email/` — Resend email integration
> - `db/schema.sql` — Postgres schema and RLS policies
>
> **Conventions:** TypeScript strict mode; Tailwind for styling; shadcn/ui component primitives in `components/ui/`; Server Components by default with explicit `"use client"` for interactivity.
>
> **Notable:** Heavy use of route groups for layout isolation; RLS policies enforced at DB layer rather than app layer.

## Tools

- `Read` and `Glob` are essential.
- `Bash` for `tree`, `wc -l`, or `git log` snippets.
- Avoid running tests or builds at this stage (slow and unnecessary for orientation).

## Failure modes

- Very large monorepos (>1000 files): summarize at directory level rather than file level.
- Mixed-language projects (Python + JS): produce two parallel summaries.
- Vendored dependencies: identify and skip.

## Related Skills

- [[match-existing-style]] (often used immediately after)
- [[refactor-typescript]]

## Changelog

### v1.0.0 — 2026-04-30: Initial creation as seed Skill.
```

```markdown
# Editorial Letter — Nonfiction

A developmental editor's first pass on a nonfiction manuscript. Focuses on structure, argument, voice, and pacing — not grammar or style at the sentence level.

## When to Use

A draft is complete and the author is ready for substantive feedback. Manuscript is at least 5000 words (otherwise the structural pass is overkill). Author wants to know what's working, what isn't, and what to do next.

## Inputs

- The manuscript file (.docx, .md, or .pdf). Full text required, not summary.
- Optional: author's stated goals for the book (audience, sales aspirations, transformation desired in the reader).

## Outputs

A markdown letter with these sections:

1. **Overall impression** — what the book is, what it's trying to do, whether it's getting there. 1–2 paragraphs.
2. **What's working** — strongest 3–5 elements. Be specific.
3. **What needs work** — biggest 3–5 structural issues, in priority order.
4. **Chapter-by-chapter notes** — one paragraph per chapter calling out specific issues.
5. **Recommended next steps** — 3–5 concrete actions the author can take.
6. **Closing** — one paragraph of perspective and encouragement.

Length: 2000–4000 words. Skipping sections is forbidden — even short sections must be present.

## Instructions

### Step 1 — Read the manuscript
End to end, no skimming. Take notes on first impressions per chapter.

### Step 2 — Identify the spine
What is the book's argument? State it in one sentence. If you can't, that's the first finding.

### Step 3 — Map the structure
Outline the manuscript in 1–2 levels: chapters and major sections. Look for: pacing imbalances, redundant chapters, missing transitions, premature conclusions.

### Step 4 — Voice and stance
Is the author's voice consistent? Does the stance shift unintentionally between chapters? Are there sections where the author hides behind passive voice or hedges?

### Step 5 — Audience match
Who is this book for? Does the writing match that audience's expertise level? Are there sections that under- or over-explain?

### Step 6 — Write the letter
Direct, specific, kind. The letter is *for the author*, not a publishing decision document. They should feel respected and energized to revise.

## Example

**Input:** A 60,000-word nonfiction manuscript on time management.

**Output (excerpt):**

> ## Overall Impression
>
> "Four Burners" is doing real work. The premise — that adults must choose which of four life burners to keep lit at any given time — is provocative, well-named, and lends itself to chapter-by-chapter elaboration. The voice is conversational without being slack, and the personal anecdotes earn their keep. At its current length, it reads as a complete book; the question is whether the structural moves earn the reader's trust through the long middle.
>
> ## What's Working
>
> - **The framing metaphor.** The four burners image is doing more work than its 1000-word introduction suggests; you can lean on it harder.
> - **Chapters 3 and 7.** These are your strongest chapters. The "burner-switching cost" framing in Chapter 3 deserves to be the book's center; consider promoting it.
> [...]
>
> ## What Needs Work
>
> - **Chapter 5 is doing two books' work.** It contains both a survey of productivity literature and a case study; pick one. The case study is stronger.
> [...]

## Failure modes

- Manuscript is fiction: this Skill is wrong; defer to a fiction-editing Skill.
- Manuscript is a draft of a draft (under 5000 words): suggest waiting for more content.
- Author is asking for line edits: redirect to a line-editing Skill.

## Tools

- `Read` (essential) for the manuscript file.
- `Write` for the editorial letter output.

## Related Skills

- [[prose-editor]]
- [[match-existing-style]] (when revisions are tonal)

## Changelog

### v1.0.0 — 2026-04-30: Initial creation as seed Skill.
```

#### Example 3 — `decision-with-tradeoffs` (DECIDE, analysis domain)
```markdown
---
name: decision-with-tradeoffs
description: |
  Produce a recommendation in the canonical decision-doc format: recommendation, reasoning, tradeoffs considered, what would change the answer. Used whenever the OPERATOR asks "should I do X" or "X or Y" and a stance with reasoning is the deliverable. Forces explicit tradeoff articulation.
version: 1.0.0
triggers:
  - should I
  - X or Y
  - what's the best approach
  - help me decide
  - recommend a path
inputs:
  - the decision context (what's at stake)
  - the options under consideration (or none, if open-ended)
  - any constraints (cost, time, risk tolerance)
outputs:
  - markdown decision doc with: Recommendation, Reasoning, Tradeoffs, Change Conditions
task_types_supported:
  - DECIDE
posture_hints:
  - primary
domain: analysis
created_at: 2026-04-30T00:00:00Z
created_by_run: seed
grounding_required: true
---

# Decision With Tradeoffs

Produces a recommendation document with explicit reasoning and tradeoffs. Forbids hedging — pick a side, state it, defend it, acknowledge what you'd give up.

## When to Use

The OPERATOR has a decision to make and wants a stance, not a survey. If they wanted "here are the options," that's an ANALYZE task; this is DECIDE.

## Inputs

- The decision context (1–2 sentences).
- Options under consideration (if predefined). If open-ended, the Skill enumerates a small set first.
- Constraints: budget, timeline, risk appetite, reversibility.

## Outputs

A markdown document with these sections:

```
# Decision: \<one-sentence framing\>
## Recommendation
\<one paragraph: pick a side and state it\>
## Reasoning
\<2–4 paragraphs: why this option, given the constraints\>
## Tradeoffs
\<list: what this recommendation gives up\>
## What Would Change the Answer
\<list: 2–4 specific conditions that would flip the recommendation\>
```javascript

## Instructions

### Step 1 — Frame the decision
Restate it in one sentence. If multiple decisions are tangled, surface that and ask which to focus on.

### Step 2 — Enumerate options (if needed)
If the OPERATOR didn't provide options, propose 2–4 distinct ones. If they did, work with what they gave.

### Step 3 — Apply the constraints
Match each option against the constraints. Eliminate non-starters explicitly.

### Step 4 — Pick
Choose one. Resist the urge to recommend "it depends" — that's not a decision.

### Step 5 — State tradeoffs
What does picking this option cost? What would the alternatives have given? List 3–5.

### Step 6 — State change conditions
What would flip the answer? List 2–4 specific things that, if true, would make a different option win.

## Example

**Input:** "Should I bootstrap Embra or raise a pre-seed round?"

**Output:**

> # Decision: Bootstrap or raise pre-seed for Embra
>
> ## Recommendation
> Bootstrap for the next 6 months while validating the law-firm GTM, then raise pre-seed against demonstrated revenue. Avoid raising on conviction alone — your conviction is high, but investors price down for speculative B2B EdTech.
>
> ## Reasoning
> [...]
>
> ## Tradeoffs
```

### 5.7 The validators
`~/cee/skill_engine/file_validators.py` runs at write time and at boot:
```python
def validate_skill_file(path: Path) -> ValidationResult:
    issues = []
    
    # 1. Slug rule
    slug = path.parent.name
    if not re.match(SLUG_REGEX, slug):
        issues.append(f"Invalid slug: {slug}")
    
    # 2. Filename
    if path.name != "SKILL.md":
        issues.append(f"File must be named SKILL.md, not {path.name}")
    
    # 3. Frontmatter parse
    try:
        frontmatter, body = parse_skill_md(path)
    except Exception as e:
        return ValidationResult(valid=False, issues=[f"Frontmatter parse failed: {e}"])
    
    # 4. Schema validate
    schema_errors = validate_against_schema(frontmatter)
    issues.extend(schema_errors)
    
    # 5. Slug consistency
    if frontmatter["name"] != slug:
        issues.append(f"frontmatter.name ({frontmatter['name']}) doesn't match directory slug ({slug})")
    
    # 6. Required body sections
    required_sections = ["When to Use", "Inputs", "Outputs", "Instructions", "Example", "Changelog"]
    for section in required_sections:
        if not has_section(body, section):
            issues.append(f"Missing required body section: {section}")
    
    # 7. Length budgets
    if frontmatter_line_count(path) > 50:
        issues.append("Frontmatter exceeds 50 lines")
    if word_count(body) > 1500:
        issues.append(f"Body exceeds 1500 words ({word_count(body)})")
    
    # 8. Changelog matches version
    if not changelog_has_version(body, frontmatter["version"]):
        issues.append(f"Changelog missing entry for v{frontmatter['version']}")
    
    # 9. Provenance for generated Skills
    if frontmatter["created_by_run"] not in ("manual", "seed"):
        if "created_from_input" not in frontmatter:
            issues.append("Generated Skills must have created_from_input")
    
    # 10. Cross-namespace collision (slug == agent slug)
    if slug in agent_registry.slugs():
        issues.append(f"Slug collision with agent: {slug}")
    
    return ValidationResult(valid=(len(issues) == 0), issues=issues)
```
The validator runs at:
- Skill generation time (before writing).
- `cee verify --skills` (manual integrity check).
- Boot's registry rebuild (warns on invalid Skills; skips them in the index).

## 8. Agent + Skill Implications
### 8.1 Cross-namespace collision is forbidden
A slug used by an agent cannot be used by a Skill, and vice versa. The validator checks both registries. This makes wiki-links unambiguous in Obsidian.
### 8.2 Skills can declare posture compatibility
The `posture_hints` field is the bridge to section 06's posture enum. The matcher uses it as a soft preference signal — `critic`-hinted Skills are slightly preferred when the AgentPlan includes a critic.
### 8.3 The `grounding_required` and `sensitivity` fields propagate up
These flags in Skill frontmatter modify `Classification.flags` when the Skill is selected. The classifier sees them via the registry; this is how a Skill can "declare a Run sensitive" even if the input doesn't trigger sensitivity patterns.

## 9. Edge Cases
**EC1 — A Skill's ****`description`**** is too short or too long.**
Validator warns at \<50 or \>300 words. Generator regenerates if outside bounds.
**EC2 — Frontmatter has typos in enum values (e.g., ****`BUIID`**** instead of ****`BUILD`****).**
Schema validation rejects.
**EC3 — A Skill's slug differs from its directory name (renamed directory but didn't update frontmatter).**
Validator catches; OPERATOR fixes one or the other.
**EC4 — Multiple Skills have nearly-identical descriptions.**
Section 07's duplicate detection catches at generation. For hand-authored Skills, validator can warn; OPERATOR consolidates manually.
**EC5 — A Skill references a deprecated tool name in the body.**
Validator can be configured with a known-tool list and warn on unknowns.
**EC6 — A Skill's body uses Markdown features Claude Code can't render.**
Body is loaded as text; rendering is a UX concern, not correctness. Validator doesn't check rendering.
**EC7 — Version bumped but changelog missing entry.**
Validator warns (configurable hard fail).
**EC8 — ****`disabled: true`**** Skill is referenced by an old Run replay.**
Replay uses `bible_snapshot` per section 03 §5.4, so old Runs continue to work.
**EC9 — Skill body contains an XML structure that looks like CEE tags.**
Body is markdown; XML in code blocks is fine. Outside code blocks, validator warns about XML-like content (could confuse the executor's role-instruction interpretation).
**EC10 — Supporting file ****`examples.md`**** has its own frontmatter.**
Allowed but not required. Frontmatter in supporting files isn't validated against the [SKILL.md](http://SKILL.md) schema; it's free-form.
**EC11 — Skill is ≤300 chars but functionally complete.**
Allowed if validators pass. Some Skills are genuinely small.
**EC12 — Hand-authored Skill marked ****`created_by_run: manual`**** but has ****`created_from_input`****.**
Allowed but unusual. Validator doesn't reject — `created_from_input` is informational.

## 10. Failure Modes
### 10.1 Schema field added but not in this page
**Failure:** `~/cee/schemas/skill_frontmatter.json` adds a field without bible update.
**Detection:** boot's cross-section consistency check (this page is the source).
**Recovery:** bible updated; schema regenerated.
### 10.2 Generator emits invalid frontmatter
**Failure:** generator's prompt drifts; output fails schema.
**Detection:** validator at write time.
**Recovery:** retry with stricter prompt; on third failure, halt with `skill_generation_failed`.
### 10.3 Length budget breach goes unnoticed
**Failure:** validator misses an oversized Skill.
**Detection:** golden Run tests that match against budget-exceeding inputs.
**Recovery:** validator strengthened; tests prevent regression.
### 10.4 Changelog drift
**Failure:** Skill version updated but changelog not.
**Detection:** validator (when enforcement enabled).
**Recovery:** OPERATOR adds entry; or generator/promotion tooling auto-adds.
### 10.5 Body section detection brittle
**Failure:** required section heading uses `H3` instead of `H2`; validator misses.
**Detection:** stricter regex; tests with edge cases.
**Recovery:** validator updated.
### 10.6 Cross-namespace collision goes undetected
**Failure:** an agent and Skill end up with the same slug.
**Detection:** boot's registry rebuild.
**Recovery:** halt with `slug_collision`; OPERATOR renames one.
### 10.7 Matcher relies on body content
**Failure:** matcher accidentally reads body instead of frontmatter, drifting matches.
**Detection:** code review; tests.
**Recovery:** matcher locked to frontmatter only.
### 10.8 Provenance lost
**Failure:** generated Skill written without `created_from_input`.
**Detection:** validator.
**Recovery:** generator fixed; back-fill from `_meta.json` if available.
### 10.9 Skill survives invalidation in index
**Failure:** invalid Skill is silently skipped at boot but cached in registry.
**Detection:** integrity check; warning in boot log.
**Recovery:** OPERATOR sees warning; fixes the Skill; next boot re-includes.
### 10.10 Changelog entries lose meaning
**Failure:** entries are vague ("updated v1.1.0").
**Detection:** OPERATOR review.
**Recovery:** validator can require minimum entry length (configurable).

## 13. Final Statement
A Skill is a contract: its frontmatter is what the matcher sees, its body is what the executor reads. The contract is small, validated, and worked-example-anchored. Every Skill in `~/cee/skills/` either follows this spec or doesn't load. Generation produces files that pass validation by construction. Hand-authored Skills inherit the same rules. The catalog stays consistent because the format is consistent — by validator, not by hope.
